# Release-Readiness Review — Polyptych Studio

Reviewed: 2026-06-14. Scope: open-source release of the public version of
`../polyptych-media-creator`.

## Verdict

**Releasable after a short cleanup pass.** Nothing here is a hard blocker, but
two items should be fixed before going public (a leaked personal domain and the
unfilled license copyright), and a handful of leftovers from the parent repo
should be tidied so the public repo looks intentional rather than copied.

What's already solid:

- All three CI gates pass locally: `ruff check src/` clean, `pyright src/` →
  0 errors, `pytest -m "not integration"` → **513 passed**.
- No secrets, API keys, or hardcoded `/Users/...` paths in tracked files.
- Clean history — single squashed "Initial commit", no leaked development
  history, no large/binary blobs (largest tracked file is `uv.lock`).
- Apache-2.0 license present; `pyproject.toml` declares it.
- The critical external dependency `pixbridge>=0.2.0` **resolves from public
  PyPI** (`uv sync` succeeds), so the public repo is self-contained.
- `.gitignore` correctly excludes `.env*`, `.envrc`, `generated/`, `output/`,
  `usage.jsonl`, and `.claude/settings.local.json`.
- A PreToolUse hook actively blocks edits to secrets-bearing files.

## Should fix before release

### 1. Personal domain leaked in a prompt template
`prompts/tasks/task-07-image-generation.md:878` hardcodes a footer in an example
image prompt:
```
- Footer: "www.dinkla.net" in small muted text, bottom-right corner
```
This is the author's personal site embedded in a shipped example. Genericize it
(e.g. `"example.com"` or drop the footer line) — otherwise generated decks may
carry it through.

### 2. License copyright owner not filled in
`LICENSE` still contains the Apache appendix placeholder
`Copyright [yyyy] [name of copyright owner]`. There is no actual copyright
notice anywhere stating owner/year. Add a real copyright line (and optionally a
`NOTICE` file). Strictly the appendix is boilerplate, but for a public release
you want an explicit `Copyright 2026 <owner>`.

## Should clean up (cosmetic, but signals "copied from parent repo")

### 3. `.gitignore` references a frontend that doesn't exist
Lines 13–18 ignore `node_modules/`, `web/node_modules/`, `web/test-results/`,
`web/coverage/`, `.vite/` — there is no `web/` dir or any JS/TS frontend in this
repo. Leftover from `polyptych-media-creator`. Remove the "Web (Vite/React
frontend)" block.

### 4. pytest marker mentions TTS — no TTS in this repo
`pyproject.toml:30` marker description: `"tests that call real external APIs
(LLM, TTS, image generation)"`. This repo has no text-to-speech. Drop `TTS`.

### 5. Two orphaned prompt files not wired into the pipeline
The code loads prompts only from `prompts/tasks/` and `prompts/providers/`
(`prompt_loader.py`). These two top-level files are not referenced by any code:
- `prompts/01-slide-analysis.md` — a **development research note** ("Analyze a
  slide presentation generated by NotebookLM…", references the essay "The
  Cellophane World" and slide images that don't ship). This is internal R&D,
  not user-facing. **Remove it** (or move to `docs/explanation/` if it has
  archival value).
- `prompts/image-gen.md` — a "Nano-Banana Pro prompting guide" reference doc,
  unlinked from anywhere. Either link it from the docs or remove it. (Note: the
  `image-gen` justfile target is unrelated — it just calls `pixbridge generate`.)

### 6. Tutorial points at a `sources/` dir that ships empty — RESOLVED
`sources/` is gitignored, so it shipped empty. Fixed by shipping
`examples/scene.md` (a short noir scene) under a non-ignored path and pointing
the README quick-start, style, and preset snippets at it (previously the
non-existent placeholder `source.md`). The tutorial now references the example
while keeping its self-contained coffee-essay heredoc for the infographic walk.

## Worth a deliberate decision (not blockers)

### 7. `.claude/` tooling ships with the repo
`.claude/skills/`, `.claude/agents/`, `.claude/hooks/`, and `.claude/settings.json`
are committed. This is intentional and documented (skills are a headline
feature). One thing to note: `settings.json` pre-allows MCP servers the public
user won't have (`mcp__backlog__*`, `mcp__plugin_context7_*`). Harmless — unknown
permission entries are simply ignored — but you may want to trim to a minimal
set so the file reads as a clean example.

### 8. `pyproject.toml` metadata is thin for a PyPI publish
No `authors`, `readme`, `[project.urls]` (homepage/repository), `keywords`, or
`classifiers`. Fine for a **source/GitHub release**; add these if you intend to
`pip install polyptych` from PyPI. Also confirm the name `polyptych` is available
on PyPI before publishing there.

### 9. Missing community/health files
No `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`, `CHANGELOG.md`, or
issue/PR templates. None are required, but a `CONTRIBUTING.md` (how to run
lint/typecheck/test) and a `SECURITY.md` (how to report) are low-effort and
expected for a public project. The README already documents the CI commands, so
CONTRIBUTING is mostly a copy-paste.

### 10. No CI badge / README polish
Optional: add a CI status badge and a one-line "Status: alpha / v0.1.0" note so
visitors know maturity.
